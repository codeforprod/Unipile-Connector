## Research Findings

### Documentation Analysis

#### Node.js Module Systems (2026 State)

**ESM (ECMAScript Modules)**:
- Modern JavaScript standard
- Uses `import`/`export` syntax
- Enabled with `"type": "module"` in package.json
- Async module loading
- Tree-shakeable

**CommonJS**:
- Node.js legacy module system
- Uses `require()`/`module.exports`
- Default in Node.js (no "type" field)
- Synchronous module loading
- Still widely used in existing projects

**Dual Package Pattern**:
- Provides both ESM and CommonJS builds
- Uses `exports` field in package.json
- Conditional exports: `import` vs `require`
- Enables compatibility across all environments

**Key Sources**:
- [Node.js Dual CommonJS/ES module packages](https://nodejs.org/api/packages.html#dual-commonjses-module-packages)
- [Supporting CommonJS and ESM with TypeScript](https://evertpot.com/universal-commonjs-esm-typescript-packages/)
- [How to build dual package npm from TypeScript](https://tduyng.com/blog/dual-package-typescript/)

**Script Breakdown**:

- `rename:cjs`: Unix `find` command to batch rename (cross-platform alternative: use node script)

**Cross-Platform Rename Alternative**:

If `find` command doesn't work on Windows, create `scripts/rename-cjs.js`:

```javascript
const fs = require('fs');
const path = require('path');

function renameJsToCjs(dir) {
  const files = fs.readdirSync(dir);
  files.forEach(file => {
    const fullPath = path.join(dir, file);
    const stat = fs.statSync(fullPath);

    if (stat.isDirectory()) {
      renameJsToCjs(fullPath);
    } else if (file.endsWith('.js')) {
      const newPath = fullPath.replace(/\.js$/, '.cjs');
      fs.renameSync(fullPath, newPath);
    }
  });
}

renameJsToCjs('dist/cjs');
```

Then update script:

### Code Patterns

#### Pattern 1: Conditional Export Configuration

```json
{
  "exports": {
    ".": {
      "import": {
        "types": "./dist/types/index.d.ts",
        "default": "./dist/esm/index.js"
      },
      "require": {
        "types": "./dist/types/index.d.ts",
        "default": "./dist/cjs/index.cjs"
      }
    }
  }
}
```

**Explanation**:
- `import` condition: Used when consumer uses `import` syntax
- `require` condition: Used when consumer uses `require()` syntax
- `types` field: TypeScript type definitions for both conditions
- `default` field: JavaScript entry point

**Order Matters**: Types before default ensures TypeScript resolves correctly.

---

#### Pattern 2: Package Markers

**dist/esm/package.json**:
```json
{
  "type": "module"
}
```

**dist/cjs/package.json**:
```json
{
  "type": "commonjs"
}
```

**Purpose**: Explicitly tells Node.js how to interpret `.js` files in each directory.

**Why Needed**: Without markers, Node.js uses parent package.json `"type"` field, which may not match the build format.

---

#### Pattern 3: Rename `.js` to `.cjs`

**Build Script**:
```bash
find dist/cjs -name '*.js' -type f -exec sh -c 'mv "$0" "${0%.js}.cjs"' {} \;
```

**Why**: TypeScript outputs `.js` files regardless of module format. Renaming to `.cjs` makes CommonJS format explicit and unambiguous.

**Alternative (Node.js script)**:
```javascript
const fs = require('fs');
const path = require('path');

function renameJsToCjs(dir) {
  fs.readdirSync(dir).forEach(file => {
    const fullPath = path.join(dir, file);
    if (fs.statSync(fullPath).isDirectory()) {
      renameJsToCjs(fullPath);
    } else if (file.endsWith('.js')) {
      fs.renameSync(fullPath, fullPath.replace(/\.js$/, '.cjs'));
    }
  });
}

renameJsToCjs('dist/cjs');
```

### API References

#### package.json Exports Field

**Specification**: [Node.js Package Entry Points](https://nodejs.org/api/packages.html#package-entry-points)

**Structure**:
```json
{
  "exports": {
    "<subpath>": {
      "<condition>": "<path>",
      "<condition>": "<path>"
    }
  }
}
```

**Common Conditions**:
- `import`: ESM import
- `require`: CommonJS require
- `types`: TypeScript definitions
- `default`: Fallback (must be last)
- `node`: Node.js specific
- `browser`: Browser environment

**Resolution Order**:
1. Node.js checks consumer's module system (ESM or CJS)
2. Looks for matching condition (`import` or `require`)
3. Resolves to specified path
4. Falls back to `default` if no match

---

#### TypeScript Module Options

**ESM Configuration**:
```json
{
  "compilerOptions": {
    "module": "NodeNext",        // Outputs ESM
    "moduleResolution": "NodeNext" // Resolves ESM imports
  }
}
```

**CommonJS Configuration**:
```json
{
  "compilerOptions": {
    "module": "CommonJS",     // Outputs CommonJS
    "moduleResolution": "Node" // Classic Node resolution
  }
}
```

**Why NodeNext**: It's TypeScript's modern module system that matches Node.js behavior exactly.

## Risks & Mitigations

### Risk 1: Build Time Increase

**Impact**: Medium
**Probability**: High

**Description**: Building two formats (ESM + CJS) will increase build time by approximately 2x.

**Mitigation**:
- Builds can be parallelized: `build:esm & build:cjs` (shell)
- Most builds are fast enough that 2x is acceptable
- Alternative: Switch to `tsup` for faster builds with esbuild

**Monitoring**: Track build times before/after to quantify impact.

---

### Risk 2: Tooling Compatibility

**Impact**: Low
**Probability**: Low

**Description**: Some older bundlers or tools may not understand modern `exports` field.

**Mitigation**:
- Keep `main`, `module`, `types` fields as fallback
- Modern Node.js (14+) and all major bundlers support `exports`
- If issues arise, can add more export conditions

**Affected Tools**: Very old webpack (< 5), Parcel (< 2), Rollup (< 2)

---

### Risk 3: Type Definition Mismatch

**Impact**: Medium
**Probability**: Low

**Description**: TypeScript types generated from ESM build may not perfectly match CJS usage.

**Mitigation**:
- Types are format-agnostic (interfaces, types don't change)
- Both builds use same types directory
- Test both formats to ensure compatibility

**Validation**: Run TypeScript compiler against test files using both formats.

---

### Risk 4: Breaking Changes for ESM Consumers

**Impact**: High (if occurs)
**Probability**: Very Low

**Description**: Refactoring exports might break existing ESM consumers.

**Mitigation**:
- ESM paths remain identical (`dist/esm/index.js`)
- `import` condition points to same files as before
- Backward compatibility is maintained by design

**Testing**: Test with existing ESM consumer before publishing.

---

### Risk 5: File Extension Issues on Windows

**Impact**: Low
**Probability**: Medium

**Description**: Unix `find` command for renaming `.js` → `.cjs` may not work on Windows.

**Mitigation**:
- Provide Node.js script alternative (`scripts/rename-cjs.js`)
- Script works cross-platform
- Document both approaches

**Affected Systems**: Windows development environments without Unix tools.
